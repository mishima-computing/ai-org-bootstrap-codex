# ADR-0005: Settledness, not dumbing — speed by shrinking the decision surface

## Status

Accepted. (Owner-ratified design principle for AI-org performance.)

## Context

The AI org's runs are slow. The common reflex is to lower the implementer's reasoning
effort: "make the implementer dumber." The owner rejects that as a lossy compression of a
real principle.

Reasoning effort is not a global throughput knob. What matters is **where** intelligence is
spent, and whether the task is truly mechanical. Dumbing the implementer pushes cognitive
load upstream onto the contract author and makes the implementer brittle. A dumb
implementer with a slightly-wrong spec produces confidently-wrong code fast, which then
costs more in the verify/revise loop. The time was not saved end-to-end; it was moved, and
may have been amplified.

## Decision

Keep every agent at full reasoning capacity. Achieve speed by shrinking the **decision
surface**, not the decider.

The guiding sentence is: make it so agents do not have to think about things that do not
need thinking (`考える必要のないことを考えなくていいようにする`). Remove settled
questions from an agent's field of view. The cheapest thought is the one nobody has to
have.

Among three org forms, adopt the middle:

- **Taylorism:** dumb implementer + smart spec. Fast when specs are perfect, brittle when
  they are not.
- **Uniformly smart:** robust, but expensive.
- **The middle (`本命の中間`):** smart agents with tight handoffs that eliminate
  **redundant** reasoning.

The middle is the organizational form of the guiding sentence.

Settledness is cashed in as removed-thought at every layer:

- **Prebuilt box image:** the environment is settled, so do not rebuild or re-derive it
  each run.
- **Parallel independent contracts:** independent tasks do not need to consider each
  other, so do not serialize them into a false shared context.
- **Frozen interface handoff:** the designer emits the API as a small frozen artifact, so
  downstream contracts receive only that instead of re-scanning the repo to re-derive it.
- **Deterministic scaffolding:** boilerplate -- package `__init__`, import stubs, test
  skeletons -- is generated by a program, not reasoned by a model.
- **Decide-once conventions:** naming, layout, and style are fixed once so they are never
  re-opened.
- **Scope as cognitive economy:** what an agent cannot see, it does not think about.

Static low effort and adaptive effort-escalation both manage the symptom: sometimes a task
is hard. Cleaning the decision surface removes the cause: settled facts sitting in the
agent's field. Once the surface is clean, the effort knob becomes moot. Full reasoning is
spent only on the genuine open residue, and the fast-wrong -> more revise loops trap is
avoided. The effort knob is the last resort, not the first.

## Consequences

- Implementation order follows the principle: do parallel + prebuilt first, because they
  remove whole classes of unnecessary thought at zero quality cost.
- Measure after those structural changes.
- Then clean the implementer's surface with deterministic scaffolding and frozen-interface
  handoffs.
- Only if a contract is still slow do we consider difficulty-proportional effort, decided
  structurally: designer-assigned risk class or failure-triggered escalation, never a
  static dumb setting.
- This matches the org's other structural choices: prefer rules and ratification over
  knobs and prohibitions.
